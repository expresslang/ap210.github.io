4 Information requirements

This clause specifies the information required for the use of the integrated resources necessary for the scope and information requirements for the design of electronic assemblies, interconnect and packaging..

The information requirements are defined using the terminology of the subject area of this application protocol.

NOTE  The information requirements correspond to those of the activities identified as being within the scope of this application protocol in Annex F.

4.1 Business concepts and terminology

This subclause describes the business context for the information required for the use of the integrated resources necessary for the scope and information requirements for the design of electronic assemblies, interconnect and packaging..

The data planning model in Figure 2 provides an overview of the information requirements of this domain.

This clause specifies the information requirements for the exchange of configuration-controlled design of parts, assemblies, interconnects, interfaces, networks, and templates. This clause specifies the information requirements for the exchange of configuration-controlled definition of requirements for those items. These items, including requirements, are considered to be products in this application protocol. The management of the configuration of those products is addressed in this application protocol, and is concerned with authorizations and the associated approvals given by persons in organizations. The structured list of materials, components, templates, or other data items needed to make a product can be exchanged, along with initial product design requirements as well as requirements for revised part versions. Directive for work to proceed on these initial or modified product versions is addressed in this application protocol.

This application protocol addresses exchange and sharing of information about the planned usage of products in a product model, and also includes data about the different products that are sold by an organization to its customers enabling an organization to control the configuration of these products. This application protocol provides the structure through which a product, unique versions of a product, and different design discipline views of a product can be defined. Several predefined views of a product that are specific to the electronics/electrotechnical industry are provided. The details of these views and other relevant information is organized in conformance options. An overview of the available conformance options is given here.

4.1.2 Electronics domain specific capabilities

Extensions to the generic capabilities for the electronics domain are provided. Due to the general nature of electronics, these options are considered applicable at higher levels of product structure as determined by application implementation.

EXAMPLE    An interconnect may be embedded in a molded or composite product, as in an automobile dashboard that is an electrotechnical assembly containing several daughterboards connected by flexible interconnect.

4.1.2.1 Network based design and analysis

Specific product structure options are provided to support network based design and analysis. Typical applications include block diagram or circuit design and analysis, with functionality provided to directly support commonly understood industrial usage with externally defined templates using languages such as Spice(TM), C(TM), VHDL-AMS(TM), SystemC(TM), Verilog(TM), or Modelica(TM). The networks may be used to represent either lumped or distributed systems depending on the approximations in the templates and the structural representation actually selected during design or analysis. The externally defined simulation template is explicitly attached to the interface of elements in the network. The networks obviously are not limited to electrical domain.

Once the functional network is accepted, physical components are selected, a process known as packaging traverses the functional hierarchy with the result that each element of the network is related to the component that implements that element. The mapping of the functional element to the physical component may be exchanged using this part of ISO 10303, as supported by the Functional decomposition to design arm. The ability of the mapping to be supported allows formal traceability to be supported between functional and physical domains supporting dynamic back annotation. The mapping results are explicitly represented by a flattened network representation of the connections between the physical components that are to be implemented by interconnect.

The network based design and analysis conformance options are:
  • Electrical network model;
  • Functional network interface model;
  • Functional network model;
  • Physical network topology model;
  • Requirement allocation to functional network element;
  • Simulation model template definition and application;
  • System wiring definition;
  • Thermal network model.

4.1.2.2 Requirements

Specific product structure options are provided to support requirements exchange, with extensive predefined classification provided. A complete capability is provided to assign quantities to a requirement. Requirements may be assigned to products, versions of products, product views and domain specific structures in those views. The results of evaluating a design against requirements, including Design Rules, may be captured. The results may include data tables, externally defined graphical representations or externally defined audio or visual representations. Traceability is provided by supporting formal assignment between a node in the requirement decomposition tree and the element in the functional or physical design that is there to satisfy that requirement. Traceability is also supported by supporting formal assignment between a node in the requirement decomposition tree and a class to which the element in the functional or physical design may be assigned, thus asserting that any element that is a member of that class is there to satisfy that requirement. The generic requirement related conformance options are:

  • Requirement;
  • Planned and evaluated characteristics.

Specific application of requirements are identified in other sections of this clause.

4.1.2.5 Packaging

Support is provided for both the assignment of mapping between the functional netlist and the assembly structure and the assignment of mapping between the functional netlist and the interconnect structure. Full support is provided for formally modeling the multi-product physical assembly needed to actually implement the detailed functional netlist. There is no limitation imposed due to assuming there is only one CAD layout application being used to implement the physical design. The physical design may be split amongst three dimensional system for complex shapes, including the usage of cables, and their connectors, and the two dimensional CAD system traditionally used for PWB board layout. There is full traceability possible through the various model instances that represent the electro-mechanical design. Principal features of this apprach include strong support for storing designs and design representations of electrical or thermal components in a library along with complete functional and simulation models for those components. In this way, a complex component will need to have its functional to physical mapping defined and stored only once. Obviously, an assembly that is physically complex can be considered a component and relevant information stored in a library. The physical interconnect "wiring" may be assigned to electrical, thermal, magnetic, or optical domains with capabilities to apply material classification properties to the "wiring" and to supporting material structures for verification purposes. The packaging conformance options are:

  • Assignment of a delay between two packaged nodes to the component implementing the path in an assembly;
  • Assignment of a packaged node to the component implementing the node in an assembly;
  • Electrotechnical assembly network model;
  • Functional network mapping to interconnect design elements.

4.1.2.7 Assembly design

The assembly design provides a similar capability to the functional design, that is hiearchical physical design is supported. The model provides the ability for a design organization to substitute various possible alternatives for analysis while maintaining the same physical interface definition. It is possible to model at the semiconductor device, package, substrate, module, black box (multiple modules in one box), drawer, rack, and system level. Parts of a design may be stored in a library as a 'macro' and re-used. Complete designs, including interfaces, may be stored in a library and re-used. Various options for physical shape are provided to support analysis. Discrete wire and detailed cable routing is supported. Critical assembly feature to feature relationships are provided to support verification that the "as-designed" assembly including interconnect satisfies the physical connectivity definition defined during packaging.

4.1.2.8 Interconnect design

As in the assembly design there is an interface definition, known as a usage view, which provides the support required to isolate the CAD model for the interconnect from that for the assembly while maintaining full traceability through the hierarchy. The connections supported by the interconnect are included in the usage view to support trade studies and test. Further, the usage view includes various options for physical shape models to support analysis. Embedded passive and active components are supported, as is various non traditional materials such as optical fiber for interconnect. Microwave models (where features of the design are considered to be components) are supported, as is the abstraction that a terminal of a component in the interconnect is on the edge of the component model rather than the surface. Full traceability of the physical netlist to the as layed out geometry is provided for planar, three dimensional and manifold surface layouts. Full support is provided for treatment of the mechanical aspects of the interconnect, including materials, stackup, and Geometric Dimensioning and Tolerancing.

4.1.2.9 Component data

This part of ISO 10303 provides complete support for all data commonly found in a component data sheet, including functional and physical properties. The physical properties, component feature classification and terminal and connection area identification are directly supported including various shape models for analysis and Geometric Dimensioning and Tolerancing. Orientation identification is unambiguously supported with direct support for both three dimensional master models as well as derived two dimensional models. Direct support for unpackaged (bare) die usage is provided, as well as packaged components. Support is provided for thermal analysis of complex multi-chip substrates such as MCM. Component features may be specified in accordance with their function, location and may be grouped. Behavioural models may be assigned to components as well as to terminals, which may be modeled as whole elements or modeled with connection areas. Connectors are an important component and there are several models provided that may be used depending on the context, ranging from a minimal connector defined to support the physical layer definition in an electrical protocol to a complete connector definition intended to support three dimensional mockup interconnect traceability. When this capability is combined with the capability to define CAD footprints for a particular package and fabrication technology, a rather complete library data set is supported. Cables are often part of an assembly or interconnect and support is provided for the external definition of the functional and physical properties of a cable, including connector positioning if available. The component data conformance options include the conformance options specified in clause 4.1.1 as well as the following:

  • Component package model in a 3D context;
  • Component package model with corresponding 2D and 3D contexts;
  • Component physical modifications;
  • Component white box model for analysis;
  • Connector model;
  • Functional specification;
  • Part feature grouping;
  • Predefined part feature functional classification;
  • Product marking;
  • Test requirement allocation;
