4 Information requirements

This clause specifies the information requirements for the Design management application module. The information requirements are specified as the Application Reference Model (ARM) of this application module.

NOTE 1  A graphical representation of the information requirements is given in Annex C.

NOTE 2  The mapping specification is specified in 5.1. It shows how the information requirements are met by using common resources and constructs defined or imported in the MIM schema of this application module.

The following EXPRESS specification begins the Design_management_arm schema and identifies the necessary external references.

EXPRESS specification:

*)
SCHEMA Design_management_arm;
(*

4.1 Required AM ARMs

The following EXPRESS interface statements specify the elements imported from the ARMs of other application modules.

EXPRESS specification:

*)
USE FROM Approval_arm;    --  ISO/TS 10303-1012

USE FROM Product_view_definition_arm;    --  ISO/TS 10303-1019

USE FROM Work_order_arm;    --  ISO/TS 10303-1043
(*

NOTE 1   The schemas referenced above are specified in the following part of ISO 10303:

Approval_arm ISO/TS 10303-1012
Product_view_definition_arm ISO/TS 10303-1019
Work_order_arm ISO/TS 10303-1043

NOTE 2   See Annex C, Figures C.1and C.2 for a graphical representation of this schema.

4.2 ARM type definitions

This subclause specifies the ARM types for this application module. The ARM types and definitions are specified below.

4.2.1 dm_approval_item   

The dm_approval_item type is an extension of the approval_item type. It adds the data type Frozen_assignment to the list of alternate data types.

NOTE  The list of entity data types may be extended in application modules that use the constructs of this module.

EXPRESS specification:

*)
TYPE dm_approval_item = EXTENSIBLE GENERIC_ENTITY SELECT BASED_ON approval_item WITH
   (Frozen_assignment);
END_TYPE;
(*

4.2.2 managed_design_object   

The managed_design_object type is an extensible list of alternate data types that allows for the designation of the data type Product_view_definition.

NOTE   The list of entity data types may be extended in application modules that use the constructs of this module.

EXPRESS specification:

*)
TYPE managed_design_object = EXTENSIBLE GENERIC_ENTITY SELECT
   (Product_view_definition);
END_TYPE;
(*

4.3 ARM entity definitions

This subclause specifies the ARM entities for this module. Each ARM application entity is an atomic element that embodies a unique application concept and contains attributes specifying the data elements of the entity. The ARM entities and definitions are specified below.

4.3.1 Add_design_object_management_relationship   

An Add_design_object_management_relationship is a type of Design_object_management_relationship.

NOTE    Together with Change_design_object_management_relationship and Delete_design_object_management_relationship, Add_design_object_management_relationship supports the exchange of changes to a design by the use of a data set containing only (1) the information necessary to identify the design, such as, an Product_view_definition or an Product_version, which is assumed to be already known to the receiving system, and (2) the design objects that have been added, changed, or deleted. All the quality and structural requirements defined in this part of ISO 10303 may also apply to the resultant new design iteration.

EXPRESS specification:

*)
ENTITY Add_design_object_management_relationship
  SUBTYPE OF (Design_object_management_relationship);
  current_design_object : managed_design_object;
UNIQUE
  UR1: current_design_object;
END_ENTITY;
(*

Attribute definitions:

current_design_object: specifies a role of the managed_design_object for the Add_design_object_management_relationship.

Formal propositions:

UR1: The current_design_object shall be unique within a population of Add_design_object_management_relationship.

4.3.2 Change_design_object_management_relationship   

A Change_design_object_management_relationship is a type of Design_object_management_relationship. The Change_design_object_management_relationship provides information about the relationship between a design object before and after a design change.

EXPRESS specification:

*)
ENTITY Change_design_object_management_relationship
  SUBTYPE OF (Design_object_management_relationship);
  current_design_object : managed_design_object;
  previous_design_object : OPTIONAL managed_design_object;
  attribute_name : OPTIONAL STRING;
WHERE
  WR1: TYPEOF(current_design_object) = TYPEOF(previous_design_object);
  WR2: current_design_object <> previous_design_object;
END_ENTITY;
(*

Attribute definitions:

current_design_object: specifies a managed_design_object which resulted from a design change.

previous_design_object: specifies a managed_design_object from which the current design was derived. The value of this attribute need not be specified.

attribute_name: specifies a string value for the Change_design_object_management_relationship. The value of this attribute need not be specified.

Formal propositions:

WR1: Type of current_design_object shall be the same type as previous_design_object.

WR2: The current_design_object should not be equal to previous_design_object.

4.3.3 Delete_design_object_management_relationship   

A Delete_design_object_management_relationship is a type of Design_object_management_relationship. Together with Change_design_object_management_relationship and Add_design_object_management_relationship, Delete_design_object_management_relationship supports the exchange of changes to a design by the use of a data set containing only (1) the information necessary to identify the design (e.g., an Product_view_definition or an Product_version), which is assumed to be already known to the receiving system, and (2) the design objects that have been added, changed, or deleted.

NOTE    Note that all the quality and structural requirements defined in this part of ISO 10303 also apply to the resultant new design iteration.

EXPRESS specification:

*)
ENTITY Delete_design_object_management_relationship
  SUBTYPE OF (Design_object_management_relationship);
  previous_design_object : managed_design_object;
UNIQUE
  UR1: previous_design_object;
END_ENTITY;
(*

Attribute definitions:

previous_design_object: specifies a role of the managed_design_object for the Delete_design_object_management_relationship.

Formal propositions:

UR1: The previous_design_object shall be unique within a population of Delete_design_object_management_relationship.

4.3.4 Design_object_management_relationship   

A Design_object_management_relationship is a type of Activity. A Design_object_management_relationship is one of an Add_design_object_management_relationship, a Change_design_object_management_relationship, or a Delete_design_object_management_relationship or additionally may be a View_dependent_design_object_management_relationship. A Design_object_management_relationship specifies the relationship between the original design and a design change via the Product_view_definition.

EXPRESS specification:

*)
ENTITY Design_object_management_relationship
  ABSTRACT SUPERTYPE OF (ONEOF (Change_design_object_management_relationship,
                                Delete_design_object_management_relationship,
                                Add_design_object_management_relationship)
                        ANDOR View_dependent_design_object_management_relationship)
  SUBTYPE OF (Activity);
WHERE
  WR1: NOT EXISTS(SELF\Activity.name);
END_ENTITY;
(*

Formal propositions:

WR1: The name shall not be populated.

4.3.5 Frozen_assignment   

A Frozen_assignment is the specification that one or more managed_design_object must be unchanged during further design activity.

EXPRESS specification:

*)
ENTITY Frozen_assignment;
  frozen_item : SET[1:?] OF managed_design_object;
INVERSE
  assigned_approval : SET[1:1] OF Approval_assignment FOR items;
END_ENTITY;
(*

Attribute definitions:

frozen_item: specifies a managed_design_object for which the design is to remain unchanged in the Frozen_assignment. There shall be one or more managed_design_object for the Frozen_assignment.

assigned_approval: specifies an inverse relationship that specifies that the existence of the Frozen_assignment is dependent on the existence of the Approval_assignment that specifies the Frozen_assignment as its items. There shall exist exactly one Approval_assignment for the Frozen_assignment.

4.3.6 View_dependent_design_object_management_relationship   

A View_dependent_design_object_management_relationship is a type of an Design_object_management_relationship.

EXPRESS specification:

*)
ENTITY View_dependent_design_object_management_relationship
  SUBTYPE OF (Design_object_management_relationship);
  previous_design : Product_view_definition;
  current_design : Product_view_definition;
END_ENTITY;
(*

Attribute definitions:

previous_design: specifies the Product_view_definition, before the change, for the View_dependent_design_object_management_relationship.

current_design: specifies the current Product_view_definition, after the change, for the View_dependent_design_object_management_relationship.



*)
END_SCHEMA;  -- Design_management_arm
